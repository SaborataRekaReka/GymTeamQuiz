# Visual Language и UI Kit для GymTeam Quiz

Документ для AI-агента, который собирает интерфейс квиз-воронки в VS Code и готовит код к переносу в Chatium.

Цель: сразу получить аккуратный, цельный и мобильный интерфейс в духе современных wellness/fitness-квизов, похожий по UX-логике на Harna/Walking Program, но без копирования чужого бренда, логотипов, графики и текстов.

## 1. Общая визуальная идея

Интерфейс должен ощущаться как лёгкий, чистый, современный фитнес-квиз для женской аудитории. Не использовать визуальный язык агрессивного инфобизнеса, казино, «успей купить», тёмных марафонов и перегруженных лендингов.

Основные ощущения:

- чистота;
- лёгкость;
- забота;
- персональность;
- доверие;
- мягкая мотивация;
- ощущение «мне сейчас помогут разобраться».

Референс по UX и визуальной логике: Harna / Walking Program quiz.
Что можно брать из референса:

- крупные карточки ответов;
- много воздуха;
- светлый фон;
- тонкий прогресс сверху;
- заметную основную кнопку;
- простые иконки;
- карточки с округлениями;
- визуальное состояние выбранного ответа;
- нижнюю sticky-кнопку на длинных экранах;
- экраны-паузы с картинкой, коротким текстом и CTA;
- аккуратную типографику.

Что нельзя копировать:

- логотип Harna;
- конкретные иллюстрации;
- английские тексты;
- уникальные графические элементы бренда;
- дословную композицию, если она выглядит как прямой клон.

## 2. Главное правило интерфейса

Один экран = одно действие пользователя.

Пользователь не должен думать, что делать дальше. На каждом шаге должно быть ясно:

- где он находится;
- какой вопрос сейчас задаётся;
- как выбрать ответ;
- как перейти дальше;
- сколько примерно осталось;
- что выбранный вариант действительно выбран.

## 3. Mobile-first

Основной дизайн делается под телефон. Десктоп вторичен.

Целевые ширины:

- 360 px: минимальная проверочная ширина;
- 390 px: основной iPhone-like сценарий;
- 430 px: крупный смартфон;
- 768 px и выше: контент остаётся в узкой колонке по центру.

Максимальная ширина основной колонки:

- 520 px для экранов вопросов;
- 560 px для результата и пейволла;
- на десктопе не растягивать карточки на всю ширину.

Контейнер:

- `width: 100%`;
- `max-width: 520px`;
- `margin: 0 auto`;
- `padding: 20px 20px 32px`;
- для экранов с sticky-кнопкой добавить нижний отступ 110–130 px.

## 4. Цветовая система

Использовать светлую тему. Тёмная тема запрещена.

Основная палитра:

```ts
const colors = {
  background: '#FFFFFF',
  surface: '#FFFFFF',
  surfaceSoft: '#F7F7FA',
  surfaceWarm: '#FFF8F5',

  text: '#2F3037',
  textMuted: '#6F717A',
  textSoft: '#9A9CA3',

  border: '#E6E7EB',
  borderStrong: '#dc2972',

  primary: '#DC2972',
  primaryHover: '#891947',
  primarySoft: '#ffeef8',

  accent: '#F27A8A',
  accentSoft: '#FFF0F2',

  success: '#2FA66A',
  successSoft: '#EAF8F1',

  warning: '#F4A340',
  warningSoft: '#FFF5E6',

  danger: '#D94A4A',
  dangerSoft: '#FFF0F0',
}
```

Правила:

- Основной акцент для кнопок и прогресса: `primary`.
- Второй акцент: `accent`, использовать дозированно для женственного wellness-ощущения, но не превращать всё в розовый интерфейс.
- Не использовать больше двух активных акцентных цветов на одном экране.
- Ошибки, успехи и предупреждения использовать только по делу.
- Фон преимущественно белый. Допускаются мягкие плашки `surfaceSoft`, `surfaceWarm`, `primarySoft`, `accentSoft`.

## 5. Типографика

Шрифт:

```css
font-family: Inter, -apple-system, BlinkMacSystemFont, 'Segoe UI', system-ui, sans-serif;
```

Если Inter недоступен в Chatium, использовать системный стек.

Размеры:

```ts
const typography = {
  h1: { fontSize: 28, lineHeight: 34, fontWeight: 700 },
  h2: { fontSize: 24, lineHeight: 30, fontWeight: 700 },
  questionTitle: { fontSize: 26, lineHeight: 32, fontWeight: 700 },
  subtitle: { fontSize: 16, lineHeight: 23, fontWeight: 400 },
  body: { fontSize: 16, lineHeight: 24, fontWeight: 400 },
  option: { fontSize: 17, lineHeight: 23, fontWeight: 650 },
  button: { fontSize: 17, lineHeight: 22, fontWeight: 700 },
  caption: { fontSize: 12, lineHeight: 17, fontWeight: 400 },
  legal: { fontSize: 11, lineHeight: 16, fontWeight: 400 },
}
```

Правила:

- Заголовки короткие, крупные, читаемые.
- Подзаголовок не длиннее 2–3 строк на телефоне.
- Варианты ответа должны читаться быстро.
- Не использовать мелкий серый текст для важной информации.
- Юридический текст может быть мелким, но ссылки должны быть видимыми и кликабельными.

## 6. Радиусы, отступы, тени

Радиусы:

```ts
const radius = {
  sm: 10,
  md: 16,
  lg: 22,
  xl: 28,
  pill: 999,
}
```

Отступы:

```ts
const spacing = {
  xs: 6,
  sm: 10,
  md: 16,
  lg: 20,
  xl: 28,
  xxl: 40,
}
```

Тени:

Использовать очень мягко. Основной интерфейс должен держаться на границах, не на тяжёлых тенях.

```ts
const shadows = {
  soft: '0 8px 24px rgba(26, 28, 35, 0.06)',
  card: '0 4px 16px rgba(26, 28, 35, 0.05)',
  sticky: '0 -8px 24px rgba(26, 28, 35, 0.08)',
}
```

## 7. Верхняя зона экрана

Компонент: `QuizHeader`.

Состав:

- слева круглая кнопка назад;
- по центру или ниже сегментированный прогресс;
- справа можно не использовать меню, если нет смысла;
- на некоторых экранах допускается маленький pill-label категории, например «Цель», «Форма», «Питание», но пользовательский текст должен быть на русском.

Важно: в ТЗ прогресс должен быть процентом, без формулировки «вопрос такой-то из такого-то». Поэтому визуальный прогресс можно делать сегментами или линией, но рядом или внутри доступно процентное значение.

Пример:

- тонкая линия прогресса 4 px;
- активная часть `primary`;
- неактивная часть `#ECEDEF`;
- подпись: `32%` или `Ваш план готов на 32%`.

## 8. Карточки ответов

Компонент: `OptionCard`.

Базовая карточка:

- высота минимум 72 px;
- ширина 100%;
- border-radius 22 px;
- border 1 px `border`;
- фон `surface`;
- padding 18–20 px;
- display flex;
- gap 14–16 px;
- align-items center;
- текст крупный и жирный;
- вся карточка кликабельная.

Состояния:

Default:

- background `#FFFFFF`;
- border `#E6E7EB`;
- text `#2F3037`.

Selected:

- background `#F7F8FF` или `#FFFFFF`;
- border 2 px `#A8AFFF` или `borderStrong`;
- subtle inner/outer shadow;
- справа check icon или заполненный чекбокс;
- можно добавить лёгкую заливку иконки.

Pressed / active:

- лёгкое уменьшение `scale(0.99)` только если не мешает производительности;
- без прыгающих эффектов.

Disabled:

- opacity 0.55;
- pointer-events none.

Ошибка:

- border `danger`;
- под карточкой короткий текст ошибки.

## 9. Single choice и multiple choice

Single choice:

- пользователь выбирает один вариант;
- можно автоматически переходить дальше через 200–350 ms, если это не ломает UX;
- если есть кнопка «Продолжить», она должна становиться активной после выбора.

Multiple choice:

- пользователь может выбрать несколько вариантов;
- обязательно нужна кнопка «Продолжить»;
- выбранные карточки должны иметь очевидное состояние;
- если ничего не выбрано, кнопка неактивна или показывает ошибку при попытке продолжить.

Для экранов с множественным выбором и большим списком использовать sticky-кнопку снизу.

## 10. Карточки с изображениями

Компонент: `ImageOptionCard`.

Использовать для вопросов, где визуальный образ помогает выбору.

Структура:

- слева изображение 72–88 px;
- border-radius 18–22 px;
- object-fit cover;
- справа текст;
- справа чекбокс или check icon.

Требования к изображениям:

- оптимизированы;
- не тяжелее разумного минимума;
- не использовать изображения, которые могут восприниматься как стыдящие тело;
- если используются фото тела, подача должна быть нейтральной и заботливой;
- изображения не должны ломать сетку на маленьких экранах.

## 11. Кнопки

Компонент: `Button`.

Primary button:

- height 56–60 px;
- border-radius pill или 22–28 px;
- background `primary`;
- color white;
- font-weight 700;
- width 100% на мобильном;
- не использовать кислотные градиенты.

Sticky CTA:

- на длинных экранах закреплять снизу;
- фон sticky-зоны белый с мягкой верхней тенью;
- padding 12–20 px 20 px 20 px;
- кнопка не должна перекрывать последний вариант ответа. Добавлять нижний padding у контента.

Secondary button:

- background `primarySoft`;
- color `primary`;
- border transparent.

Ghost / back:

- круглая кнопка 36–40 px;
- background `surfaceSoft`;
- icon color `text`.

## 12. Инпуты

Компонент: `InputField`.

Используется для:

- веса;
- роста;
- целевого веса;
- имени;
- почты.

Визуал:

- height 56–60 px;
- border-radius 18–22 px;
- border 1 px `border`;
- padding 0 16 px;
- font-size 17 px;
- focus border `primary`;
- error border `danger`.

Для веса/роста:

- справа единица измерения: `кг`, `см`;
- ввод только чисел;
- ограничить очевидно неверные значения через валидацию.

Для почты:

- проверять базовый формат email;
- если ошибка, показывать короткое сообщение: `Введите корректную почту`.

## 13. Чекбокс согласия

Компонент: `ConsentCheckbox`.

Требования:

- показывается только на лид-гейте;
- без чекбокса нельзя открыть результат;
- текст короткий, ссылки кликабельные;
- ссылки: оферта, политика конфиденциальности, согласие на обработку персональных данных;
- текст мелкий, но читаемый.

Визуал:

- checkbox 22–24 px;
- selected state `primary`;
- рядом legal text 11–12 px;
- строки не должны слипаться.

## 14. Экраны доверия

Компонент: `TrustScreen`.

Структура:

- изображение сверху или карточка-иллюстрация;
- крупный заголовок;
- короткий текст;
- кнопка «Продолжить»;
- много воздуха.

Важно:

- экран должен ощущаться как пауза и усиление доверия;
- не перегружать;
- не использовать огромные абзацы;
- если есть цифра, выделить её аккуратно.

## 15. Лоадеры

Компонент: `LoaderScreen`.

Лоадер должен усиливать ощущение персонализации.

Структура:

- заголовок;
- прогресс или мягкая анимация;
- список подпунктов, которые последовательно подсвечиваются;
- без агрессивных спиннеров и мигания.

Пример подпунктов:

- `Подбираем тренировки под ваши зоны`;
- `Считаем калории и норму питания`;
- `Рассчитываем, когда вы увидите результат`.

Длительность искусственного ожидания:

- не больше 2–3 секунд;
- если платформа тормозит, не добавлять лишнюю задержку.

## 16. Лид-гейт

Компонент: `LeadGateScreen`.

Цель: собрать имя, почту и согласие перед показом результата.

Структура:

- заголовок: план почти готов;
- короткое объяснение;
- поле имени;
- поле почты;
- чекбокс согласия;
- кнопка открытия результата.

Важно:

- не писать «пришлём на почту», потому что по ТЗ письмо не отправляется;
- почта нужна только для открытия результата и сохранения лида;
- если поля не заполнены, показывать понятные ошибки.

## 17. Экран результата

Компонент: `ResultScreen`.

Цель: создать ощущение, что план реально персональный.

Структура:

1. Персональный заголовок:
   `{Имя}, вот ваш план: с {вес} до {целевой вес}`.

2. Визуализация «сейчас и после»:
   - не делать стыдящую визуализацию;
   - использовать мягкую и нейтральную подачу;
   - можно показывать силуэт, прогресс-карту или карточки параметров.

3. Сводка ответов:
   - цель;
   - зоны;
   - формат тренировок;
   - опыт;
   - мотивация.

4. Метрики:
   - ИМТ;
   - норма калорий;
   - КБЖУ;
   - разница до цели;
   - ориентир срока.

5. Подборка программ:
   - 2–4 карточки программ;
   - каждая карточка объясняет, почему подходит.

6. CTA к пейволлу:
   - `Открыть доступ к программам` или аналогичный русский текст из логики проекта.

## 18. Шкала ИМТ

Компонент: `BmiScale`.

Визуал:

- горизонтальная шкала;
- мягкие цветовые зоны;
- маркер текущего значения;
- подпись текущего ИМТ;
- без медицинского давления и диагнозов.

Текст:

- аккуратный, без запугивания;
- не обещать медицинский результат.

## 19. Пейволл

Компонент: `PaywallScreen`.

Пейволл должен быть продающим, но не крикливым.

Обязательные блоки:

1. Оффер от результата человека.
2. Социальные доказательства про Катю Усманову.
3. Визуализация результата.
4. Состав подписки под цель человека.
5. Тарифы:
   - Лёгкий старт: 390 руб, зачёркнуто 590, 7 дней, дальше автопродление 1490 руб в месяц;
   - Популярно: 1490 руб, зачёркнуто 2990, 1 месяц;
   - Хит продаж: 4990 руб, зачёркнуто 9990, 6 месяцев.
6. Гарантии и значки безопасности.
7. FAQ.
8. Эмбед формы ГетКурса.
9. Юридический футер.

## 20. TariffCard

Компонент: `TariffCard`.

Визуал:

- карточки тарифов вертикально на мобильном;
- выбранный/рекомендуемый тариф выделять мягко;
- бейджи `Популярно`, `Хит продаж`;
- цена крупная;
- зачёркнутая старая цена рядом или выше;
- условия автопродления видимы, не прятать мелко.

Не использовать:

- `осталось N мест`;
- мигающие таймеры;
- давление и фальшивый дефицит.

## 21. FAQ

Компонент: `FaqItem` или `Accordion`.

Требования:

- вопросы в одну колонку;
- раскрытие плавное, но без тяжёлой анимации;
- текст короткий;
- ответы закрывают возражения: автопродление, как отключить подписку, кому подходит, что внутри, как работает помощник по питанию.

## 22. Иконки

Иконки должны быть:

- простые line-icons;
- 20–24 px;
- цвет `primary` или `textMuted`;
- без сложных 3D-иконок;
- единый стиль толщины линий.

Для MVP можно использовать emoji только как временный placeholder в коде, но в финальной версии лучше заменить на SVG/line icons.

## 23. Фото и визуальные блоки

Фото Кати использовать на экранах доверия и пейволле.

Правила:

- фото не должны быть тяжёлыми;
- использовать с округлениями;
- не ставить слишком много фото подряд;
- фото должно усиливать доверие, не забивать интерфейс;
- не размещать логотипы и ссылки Instagram / Meta.

## 24. Анимации

Разрешены:

- мягкое появление карточек;
- подсветка выбранного ответа;
- плавное раскрытие FAQ;
- спокойный лоадер.

Запрещены:

- мигающие элементы;
- прыгающие кнопки;
- агрессивные shake-анимации;
- параллакс;
- тяжёлые Lottie без необходимости;
- всё, что может ухудшить скорость загрузки.

## 25. Состояния ошибок

Ошибки должны быть спокойными и полезными.

Примеры:

- `Выберите хотя бы один вариант`;
- `Введите вес в килограммах`;
- `Введите рост в сантиметрах`;
- `Введите корректную почту`;
- `Поставьте галочку согласия, чтобы открыть результат`.

Не использовать обвиняющий тон.

## 26. Доступность

Минимальные требования:

- кликабельные зоны не меньше 44 px;
- цветовой контраст достаточный;
- текст не меньше 16 px для основного контента;
- интерактивные элементы имеют focus-state;
- выбранное состояние понятно не только цветом, но и иконкой/бордером;
- кнопки и карточки имеют понятные aria-label, если это применимо в выбранной реализации.

## 27. Технические ограничения UI

Не подключать:

- Material UI;
- Ant Design;
- Bootstrap;
- Mantine;
- Chakra UI;
- полноценный shadcn/ui пакет;
- Radix UI без крайней необходимости;
- тяжёлые animation libraries.

Можно использовать:

- собственные TypeScript UI-примитивы;
- shadcn/ui как визуальный и архитектурный референс;
- простые CSS tokens;
- простые SVG-иконки;
- минимальные CSS transitions.

## 29. Принцип разработки компонентов

Каждый компонент должен быть:

- маленьким;
- читаемым;
- без лишней магии;
- с явными props;
- переиспользуемым;
- легко переносимым в Chatium;
- с понятными состояниями.

Не делать один огромный компонент `QuizApp`, в котором лежит весь интерфейс.

## 30. Визуальные критерии приёмки

Перед сдачей проверить:

- все экраны выглядят как часть одной системы;
- одинаковые карточки имеют одинаковые радиусы, отступы и состояния;
- кнопка на всех экранах одинаковая;
- прогресс одинаковый;
- selected-state очевиден;
- на мобильном ничего не обрезается;
- sticky-кнопка не перекрывает варианты ответа;
- поля ввода удобны;
- лид-гейт выглядит доверительно;
- результат выглядит персональным;
- пейволл не выглядит как отдельный чужой лендинг;
- юридический футер есть;
- интерфейс не перегружен изображениями;
- нет английских пользовательских текстов;
- нет тёмной темы;
- нет тяжёлых анимаций.

## 31. Короткая инструкция для AI-агента

Если нужно принять визуальное решение, выбирай:

- меньше элементов;
- больше воздуха;
- крупнее кликабельные зоны;
- мягче цвета;
- понятнее состояния;
- быстрее загрузку;
- более цельный стиль.

Если сомневаешься между красивым и понятным, выбирай понятное.
Если сомневаешься между эффектным и быстрым, выбирай быстрое.
Если сомневаешься между сложным и поддерживаемым, выбирай поддерживаемое.
